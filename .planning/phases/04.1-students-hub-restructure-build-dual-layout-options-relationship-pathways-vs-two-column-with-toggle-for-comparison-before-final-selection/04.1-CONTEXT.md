# Phase 04.1: Students Hub Restructure - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Explore two alternative layout structures for the Students Hub to move away from the Current/Prior/Prospective student lifecycle categorization. Build both layouts and provide a comparison mechanism (toggle/side-by-side view) so the user can test and decide which approach better serves the expanded student content.

**Key Objective:** Determine the most effective information architecture for organizing student resources around topics and engagement opportunities rather than student lifecycle stages.

**Expanded Content to Accommodate:**
- Getting Started information (office location, communication preferences, how to reach out, mentorship philosophy, first conversation, what you provide, fit assessment)
- Collaboration opportunities (research, TA, curriculum development)
- Professional support (grad school recommendation letters, job references)
- Student outcomes and impact (testimonials, success stories, poster presentations from collaborations)

**Success Criteria:**
1. Both layouts built and fully functional
2. User can easily toggle/compare between Layout 1 and Layout 2
3. All 4 core topics accessible in each layout
4. Responsive on mobile (no Current/Prior/Prospective tabs)
5. User can make a final selection for Phase 5 (Polish & Verification)

</domain>

<decisions>
## Implementation Decisions

### Core Topics (Replaces Current/Prior/Prospective)

All content organized around these 4 topics instead of student lifecycle stages:

1. **Get Started** (Onboarding & Orientation)
   - Office location and hours
   - Communication preferences (email, office hours, response time expectations)
   - What to include in initial email (resume, interests, background, etc.)
   - Availability windows (semester-based, year-round, summer availability)
   - Mentorship philosophy (how you work with students, your approach)
   - Typical first conversation (what will happen when they reach out)
   - What you provide (guidance, mentorship, connections, publications, recommendation letters, funding, etc.)
   - Fit assessment (how you decide if student is good match for collaboration)

2. **Collaboration** (Engagement Opportunities)
   - Research opportunities (ongoing projects, areas of focus, requirements)
   - TA opportunities (in which courses, how to apply, expectations)
   - Curriculum development projects (past examples like ECE 241, how to get involved)

3. **Professional Recommendations** (Graduate School & Career Support)
   - Recommendation letters for graduate school (how to request, timeline, requirements, process)
   - Job application references (how to include you as a reference, what to provide, timeline)

4. **Student Impact/Outcomes** (Testimonials & Success Stories)
   - Testimonials from students (thank you messages, stories of impact)
   - Success stories (students admitted to grad school, got jobs, career progression)
   - Poster presentations from research collaborations (showcase of student work outcomes)

### Layout 1: Two-Column Sidebar Navigation

**Structure:**
- Left sidebar: Navigation list showing all 4 topics (always visible)
- Right content area: Full content for selected topic
- Topic list is the primary navigation mechanism

**Desktop Behavior:**
- Sidebar fixed on left (sticky)
- Content area on right, scrolls independently
- All topics visible at all times
- User clicks topic name to load that section's content
- Current/selected topic visually highlighted in sidebar

**Mobile Behavior (Dropdown/Select Menu):**
- Topics hidden in a dropdown/select menu at top of page
- User taps dropdown, selects topic
- Content displays below dropdown
- Compact, vertical layout
- No horizontal scrolling
- Sidebar becomes a select menu due to screen size

**Mobile Technical Implementation:**
- On screens < 768px: Convert sidebar to HTML `<select>` or dropdown button + list
- On screens ≥ 768px: Display sidebar normally
- JavaScript to handle topic switching in both modes
- Same content, different navigation mechanism

**Advantages:**
- All topics always visible (on desktop)
- Fast topic switching without scrolling
- Familiar information architecture pattern
- Clean visual separation between navigation and content
- Scalable (easy to add more topics later)

**Disadvantages:**
- Sidebar takes horizontal space on desktop
- Dropdown on mobile is less discoverable (topics hidden by default)
- Requires JavaScript for mobile dropdown

### Layout 2: Expandable Cards

**Structure:**
- Full-width page with topic cards stacked vertically
- Each card shows: topic title + brief description/preview
- User clicks card or expand button to reveal full content
- Full content can expand below card or as modal (to be decided during planning)

**Desktop Behavior:**
- Cards displayed in single column or 2-column grid (to be tested)
- Each card shows title + 1-2 sentence summary
- Click to expand → reveals full content below card or in place
- All cards visible at once for quick scanning
- Visual consistency with existing card components from Phase 1-3

**Mobile Behavior:**
- Single column stack
- Same expand/collapse mechanism
- Touch-friendly card targets (44px+ height)
- Natural scrolling experience

**Advantages:**
- All topics visible at glance (high discoverability)
- Natural scrolling experience
- Card format consistent with existing design system
- Scannable — users can skim topic names
- Less JavaScript required (just expand/collapse)
- Familiar pattern (accordion-like)

**Disadvantages:**
- More vertical space on page
- Scrolling required to see all topics initially
- Expanded cards push content down

### Comparison Mechanism

**How to implement toggle/comparison:**
- Provide a toggle switch at top of page: "Layout 1" ↔ "Layout 2"
- Toggle persists for session (localStorage) so user can switch back/forth
- Both layouts render from same content source (avoid duplication)
- User can A/B test and decide which feels better

**Alternative:** Side-by-side comparison on desktop (both layouts displayed simultaneously) for direct comparison. Mobile shows single layout with toggle.

### Content Organization Within Each Layout

**Same 4-topic structure in both layouts:**
- Get Started always first (most relevant for new students)
- Collaboration, Professional Recommendations, Student Impact/Outcomes follow
- Order doesn't change between layouts
- Content is identical; only presentation differs

### Carry-Forward from Prior Phases

**Page Header:** Navy background section with H1 "Students Hub" and lead text — consistent with Phase 3 pattern
- Min-height: 35-40vh
- Typography: Georgia serif H1, white text on navy
- Lead text: Updated to reflect new content focus

**Card Styling:** Yellow top border cards for important content sections (Recommendation Letters card, etc.) — consistent with existing `.card` and `.card--highlighted` classes

**Responsive Design:**
- Breakpoints: 640px (mobile→tablet), 768px (nav), 1024px (desktop)
- Both layouts adapt gracefully to all breakpoints
- Desktop: Full layout optimized for wider screens
- Tablet: Intermediate layout (sidebar may adjust or convert)
- Mobile: Optimized single-column experience

**Typography & Styling:**
- All tokens from Phase 1 design system
- H1/H2: Georgia serif
- Body: Inter sans-serif
- Colors: Navy, yellow accents, Meliora Blue (hover states)
- All spacing tokens: `--spacing-xs` through `--spacing-3xl`

**Animations:**
- Page-load fade-in for major sections (wrapped in `@media (prefers-reduced-motion: no-preference)`)
- Smooth expand/collapse transitions for expandable content
- Consistent 0.2s transition timing

**Accessibility:**
- Full ARIA support for both layouts
- Keyboard navigation (Tab, Arrow keys, Enter to expand/collapse)
- Focus indicators visible on all interactive elements
- Semantic HTML (`<main>`, `<section>`, `<button>`, `aria-expanded`, etc.)
- Screen reader support: content order logical, labels clear

**First-Person Voice:**
- All copy must use "I/my/me" (core project requirement from CLAUDE.md)
- Consistent with Phase 1-4 content

- Select menu styling vs dropdown button + list for Layout 1 mobile
- Card padding, margins, spacing within each layout
- Precise wording of "Get Started" section content (exact descriptions of mentorship philosophy, fit assessment criteria, etc.)
- Which layout is shown first/by default

</decisions>

<specifics>
## Specific Ideas & Preferences

- Both layouts should feel modern, clean, and professional — matching the faculty website aesthetic
- "Get Started" should feel welcoming and set expectations clearly for new students
- The toggle/comparison should be easy and obvious — users should feel comfortable switching between layouts multiple times
- Visual consistency: Both layouts use existing card components and design tokens from Phase 1
- The goal is not just aesthetic preference, but usability — which layout helps students find what they need faster?
