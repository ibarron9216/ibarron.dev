## Summary

Phase 04.1 rebuilds `students.html` from the existing three-tab lifecycle interface (Current/Prior/Prospective) into two fully functional alternative layouts organized around four topic areas (Get Started, Collaboration, Professional Recommendations, Student Impact/Outcomes). The primary technical challenge is implementing two fundamentally different UX patterns in a single HTML file without content duplication, then providing a clean toggle mechanism so the user can experience and compare both layouts before choosing one for Phase 5.

The project's vanilla-JS-only constraint is well-suited to both patterns. Layout 1 (sidebar navigation) requires a CSS Grid sticky sidebar with JavaScript-driven content switching, borrowing the same `switchTab()`-style pattern already in `js/main.js`. Layout 2 (expandable cards) follows the ARIA accordion disclosure pattern from the W3C Authoring Practices Guide, which is simpler JavaScript than the existing tab system. The most nuanced technical element is the layout toggle itself: a `role="switch"` button with `aria-checked`, localStorage persistence, and a data attribute on a wrapper element (`data-layout="1"` / `data-layout="2"`) that CSS uses to show/hide each layout.

The existing codebase is in excellent shape: `initTabs()` in `js/main.js` already handles `transitionend`-based fade animations, `tabindex` roving focus, and prefers-reduced-motion guards. These exact patterns transfer directly to both new layouts.

**Primary recommendation:** Use a single `data-layout` attribute on `<main>` (or a `.students-layouts` wrapper), toggled by JavaScript, to switch between the two rendered layouts with CSS `display: none` / `display: block`. Both layouts live in the same DOM. Content is written once and placed inside each layout's markup (the 4 topic texts are short enough that duplication in HTML is acceptable and far simpler than any content-injection approach).

### Pattern 1: Layout Toggle with data-attribute + localStorage

**What:** A `<button role="switch" aria-checked="false">` at the top of the page toggles `data-layout="1"` / `data-layout="2"` on the `.students-content` wrapper. CSS uses `[data-layout="2"] .students-layout-1 { display: none; }` to show/hide each layout. On page load, JavaScript reads `localStorage.getItem('studentsLayout')` to restore the last choice.

**When to use:** Any time you need to switch between two full DOM trees without rebuilding them. The data-attribute pattern keeps layout logic in CSS where it belongs; JS only writes state.

**Example:**

```javascript
// Source: W3C APG role="switch" pattern + localStorage pattern
function initLayoutToggle() {
  const toggle = document.querySelector('.layout-toggle__btn');
  const content = document.querySelector('.students-content');
  if (!toggle || !content) return;

  // Restore from localStorage
  const saved = localStorage.getItem('studentsLayout') || '1';
  setLayout(saved);

  toggle.addEventListener('click', () => {
    const current = content.getAttribute('data-layout');
    const next = current === '1' ? '2' : '1';
    setLayout(next);
    localStorage.setItem('studentsLayout', next);
  });

  function setLayout(layoutNum) {
    content.setAttribute('data-layout', layoutNum);
    toggle.setAttribute('aria-checked', layoutNum === '2' ? 'true' : 'false');
    // Update visible label text
    const label = toggle.querySelector('.layout-toggle__label');
    if (label) label.textContent = layoutNum === '1' ? 'Layout 1: Sidebar' : 'Layout 2: Cards';
  }
}
```

```css
/* Source: CSS data-attribute + display toggle */
.students-layout-1,
.students-layout-2 {
  display: none;
}
[data-layout="1"] .students-layout-1 { display: block; }
[data-layout="2"] .students-layout-2 { display: block; }
```

### Pattern 2: Layout 1 Sidebar Navigation

**What:** CSS Grid two-column wrapper. Left column: `<nav>` with a `<ul>` of 4 topic buttons (not links). Right column: 4 content panels, only one visible at a time (same hidden-attribute pattern as existing tabs). JavaScript handles click and keyboard switching via an adapted `switchTab()`.

**When to use:** When all navigation items must be visible simultaneously on desktop, and content is too dense for scrolling.

**CSS Grid structure (confirmed by research):**

```css
/* Source: CSS Grid sticky sidebar — verified pattern */
.layout-1__wrapper {
  display: grid;
  grid-template-columns: 240px 1fr;
  gap: var(--spacing-2xl);
  align-items: start; /* CRITICAL: required for sticky to work in grid */
}

.layout-1__sidebar {
  position: sticky;
  top: calc(64px + var(--spacing-lg)); /* nav height + buffer */
  /* NOTE: height is NOT set here — we want the sidebar to be full natural height */
  /* The sticky behavior only needs top; height:100vh is only for overflow-y:auto scrollable sidebars */
}

@media (max-width: 767px) {
  .layout-1__wrapper {
    grid-template-columns: 1fr; /* Stack on mobile */
  }
  .layout-1__sidebar { display: none; } /* Hidden on mobile */
  .layout-1__mobile-select { display: block; } /* Select visible on mobile */
}

@media (min-width: 768px) {
  .layout-1__mobile-select { display: none; } /* Hidden on desktop */
}
```

**ARIA for sidebar navigation list:**

```html
<!-- Source: W3C APG navigation landmark + button pattern -->
<nav class="layout-1__sidebar" aria-label="Topic navigation">
  <ul role="list" class="sidebar-nav__list">
    <li>
      <button
        type="button"
        class="sidebar-nav__item"
        aria-selected="true"
        data-controls="topic-get-started"
      >Get Started</button>
    </li>
    <!-- 3 more items -->
  </ul>

```

NOTE: Do NOT use `role="tablist"` + `role="tab"` for the sidebar. That ARIA pattern requires arrow-key navigation (roving tabindex), which is awkward in a vertical sidebar. Use regular `<button>` elements in a `<nav>`, navigable with Tab, with `aria-selected` state for active indication. This is the standard navigation landmark pattern.

**Mobile: native `<select>` (confirmed recommendation):**

### Pattern 3: Layout 2 Expandable Cards (ARIA Accordion)

**What:** 4 `.topic-card` elements, each containing an `<h2>` with a `<button aria-expanded>` inside, and a sibling content `<div>`. Enter/Space expand/collapse. Max-height CSS transition for smooth open animation.

**ARIA structure (from W3C APG accordion example — HIGH confidence):**

```html
<!-- Source: https://www.w3.org/WAI/ARIA/apg/patterns/accordion/ -->
<div class="topic-card">
  <h2 class="topic-card__heading">
    <button
      type="button"
      class="topic-card__trigger"
      aria-expanded="false"
      aria-controls="topic-content-get-started"
      id="topic-btn-get-started"
    >
      <span class="topic-card__title">Get Started</span>
      <span class="topic-card__icon" aria-hidden="true">+</span>
    </button>
  </h2>
  <div
    id="topic-content-get-started"
    role="region"
    aria-labelledby="topic-btn-get-started"
    class="topic-card__content"
    hidden
  >
    <!-- full topic content -->
  </div>
</div>
```

**CRITICAL ARIA NOTE:** The W3C APG says to use `role="region"` on the content panel only when there are 6 or fewer panels (to avoid landmark proliferation). With 4 panels, `role="region"` + `aria-labelledby` is correct and recommended.

**Expand/collapse animation — recommended approach (max-height, avoids height:auto pitfall):**

```css
/* Source: CSS-Tricks "Using CSS Transitions on Auto Dimensions" */
/* height:auto cannot be animated — use max-height approach */
@media (prefers-reduced-motion: no-preference) {
  .topic-card__content {
    overflow: hidden;
    max-height: 0;
    transition: max-height 0.3s ease-out;
  }

  .topic-card__content.is-open {
    max-height: 800px; /* Large enough for any topic content */
    transition: max-height 0.3s ease-in;
  }
}
```

JavaScript toggles:
1. `aria-expanded="true"/"false"` on the button
2. `hidden` attribute removal/addition on the content div
3. `.is-open` class on the content div (for max-height animation)

NOTE: Do NOT animate height directly. `height: auto` cannot be CSS-transitioned. The `max-height` approach with a generous ceiling (800px) is the established workaround for content-sized elements. The timing mismatch is imperceptible at this scale.

### Pattern 4: The Toggle Switch UI

**What:** A `<button role="switch" aria-checked="false">` near the top of the page, before the main content area. Styled to look like a toggle/pill switch.

**ARIA (from Erik Kroes' accessible toggle guide — HIGH confidence):**

```html
<!-- Source: https://www.erikkroes.nl/blog/accessible-html-toggle-button-switch/ -->
<div class="layout-toggle">
  <span class="layout-toggle__label-text" id="toggle-label">View layout:</span>
  <button
    type="button"
    class="layout-toggle__btn"
    role="switch"
    aria-checked="false"
    aria-labelledby="toggle-label"
  >
    <span class="layout-toggle__option" aria-hidden="true">Sidebar</span>
    <span class="layout-toggle__option" aria-hidden="true">Cards</span>
  </button>
</div>
```

CSS uses `[aria-checked="false"]` and `[aria-checked="true"]` selectors to style the active option visually. This satisfies WCAG 4.1.2 (Name, Role, Value) without relying on color alone.

### Anti-Patterns to Avoid

- **Using `role="tablist"` for the sidebar:** The tab pattern requires arrow-key navigation (roving tabindex), which is unnatural in a vertical nav list where Tab is expected. Use `<nav>` + `<button>` with standard Tab navigation instead.
- **Using `display: none` to hide layout 1 vs layout 2 alternatives mid-switch:** Instead, use `data-layout` on the wrapper and CSS `display` toggling. Never modify individual panel visibility from the toggle — that causes state conflicts.
- **Animating `height` directly:** `height: auto` cannot be CSS-transitioned. Always use `max-height` with a generous ceiling value for accordion animation.
- **Caching `prefersReducedMotion`:** The existing code calls it as a function `prefersReducedMotion()` each time, which is correct. Do not cache the result — it must reflect the current OS setting.
- **`role="region"` on more than 6 panels:** With 4 panels, this is safe. If more panels are added in Phase 5+, remove region roles to avoid landmark pollution.
- **Writing topic content twice in HTML:** For 4 topics with modest content, accepting mild HTML repetition between the two layouts is simpler and more maintainable than JS content injection. Both layouts get full markup. The toggle hides the entire layout block, not individual content nodes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Animated expand/collapse | Custom `scrollHeight` JS measurement | CSS `max-height` transition | 27+ lines of brittle JS vs. 6 lines of CSS; JS approach breaks on padding changes |
| Accessible toggle | Custom checkbox + label hack | `<button role="switch" aria-checked>` | Native button handles Enter/Space; role="switch" is the correct semantic |
| Mobile topic dropdown | Custom dropdown widget with ARIA listbox | Native `<select>` element | Fully keyboard-accessible by browser, no ARIA custom implementation needed, works on all mobile browsers |
| Layout persistence | Cookie parsing | `localStorage.getItem/setItem` | 2 lines vs. 30; zero dependencies; already used in the existing codebase pattern |
| Sticky sidebar | JS scroll listener + `position: fixed` | `position: sticky` + `align-items: start` on grid | CSS sticky is hardware-accelerated; JS polling causes jank |

**Key insight:** The pattern that removes the most code is also the most accessible. Native HTML elements (`<select>`, `<button>`, `<h2>`) already carry keyboard behavior and ARIA semantics. Custom implementations almost always miss edge cases.

## Common Pitfalls

### Pitfall 1: `position: sticky` Failing in CSS Grid

**What goes wrong:** The sidebar appears to scroll with the page instead of sticking.

**Why it happens:** `position: sticky` requires the containing block to have `overflow: visible` (the default). CSS Grid items also need `align-items: start` on the grid container — if `align-items` defaults to `stretch`, the grid item stretches to the full grid height and has nowhere to stick within.

**How to avoid:** Always add `align-items: start` to the grid container (`.layout-1__wrapper`). Also, do NOT set `overflow: hidden` or `overflow: auto` on any ancestor between the sticky element and the viewport.

**Warning signs:** Sidebar sticks on the live demo but not in the project — check for `overflow: hidden` on `.main-content`, `.section`, or `.container`.

### Pitfall 2: `height: auto` Transition Does Nothing

**What goes wrong:** The expand/collapse animation doesn't animate — content appears/disappears instantly.

**Why it happens:** CSS cannot interpolate between `0` and `auto`. This is a fundamental CSS limitation, not a browser bug.

**How to avoid:** Use `max-height: 0` (collapsed) and `max-height: 800px` (open) with `overflow: hidden`. The `800px` ceiling is never reached by the content, so the transition runs at natural content speed.

**Warning signs:** Developer tools show the transition applied but no visual animation. The fix is switching `height` to `max-height`.

### Pitfall 3: Layout Toggle Breaks Existing Tab Initialization

**What goes wrong:** `initTabs()` finds `[role="tablist"]` in the DOM and operates on it — but there is no longer a tablist in the new students.html. Alternatively, Layout 1's sidebar buttons pick up unintended tab-related CSS.

**Why it happens:** `students.html` is being completely rewritten; `initTabs()` in `main.js` currently assumes the tab pattern. When the tablist is gone, `initTabs()` returns early and does nothing — this is safe. But if Layout 1 sidebar buttons accidentally get `[role="tab"]` (to reuse CSS), the tab CSS applies globally.

**How to avoid:** Do NOT assign `role="tab"` to sidebar buttons. Use `.sidebar-nav__item` as the class (not `[role="tab"]`). In `main.js`, `initTabs()` will gracefully no-op (already has `if (!tablist) return;` guard). The new functions `initSidebarNav()` and `initExpandableCards()` are separate and only activate if their respective DOM elements exist.

**Warning signs:** Sidebar buttons get the navy pill style from the existing `[role="tab"]` CSS rule in Phase 4 section of `style.css`.

### Pitfall 4: Both Layouts Visible Briefly on Page Load (Flash)

**What goes wrong:** Before JavaScript runs, both layout blocks are visible (no `data-layout` set yet on the wrapper), causing a flash of both layouts.

**Why it happens:** The `data-layout` attribute is set by JavaScript on `DOMContentLoaded`. Between HTML parse and JS execution, the wrapper has no attribute, so CSS shows nothing (or shows both if the default is set wrong).

**How to avoid:** Set `data-layout="1"` directly in the HTML as a default. JavaScript overrides it on DOMContentLoaded if localStorage has a different value. This ensures the correct layout is visible immediately from HTML, with no flash.

```html
<!-- students.html — default in HTML prevents flash -->
<div class="students-content" data-layout="1">
```

### Pitfall 5: Mobile Select Out of Sync with Content Panels

**What goes wrong:** The mobile `<select>` shows topic A, but the visible content panel is topic B (from a prior desktop interaction).

**Why it happens:** The sidebar nav buttons and the mobile select both control the same content panels, but they are independent UI elements. If someone sizes from desktop to mobile mid-session, the select may not reflect the currently active topic.

**How to avoid:** In `showTopic(topicId)` (the shared switching function), always sync both the sidebar buttons' `aria-selected` state AND the select's `value` property:

```javascript
function showTopic(topicId) {
  // Show/hide panels (same as switchTab pattern)
  // Update sidebar button aria-selected
  // Sync mobile select value
  const mobileSelect = document.querySelector('.layout-1__select');
  if (mobileSelect) mobileSelect.value = topicId;
}
```

### Pitfall 6: Accordion `aria-expanded` Not Announced After First Toggle

**What goes wrong:** Screen readers announce the first expand correctly, but subsequent toggles are not announced.

**Why it happens:** Some screen readers cache button state. This usually occurs when the button's accessible name changes on toggle (e.g., from "Expand: Get Started" to "Collapse: Get Started") but the DOM text update happens outside the button element.

**How to avoid:** Keep the button text static (the topic title). Put `aria-expanded` on the button. Let the screen reader announce the state change via `aria-expanded` — do not additionally change the button's text content on toggle. Use a visually-hidden status element only if testing shows a gap.

## Code Examples

### JavaScript: showTopic() — shared by both sidebar and mobile select

```javascript
// Adapted from initTabs() switchTab() pattern in js/main.js
function initSidebarNav() {
  const sidebar = document.querySelector('.layout-1__sidebar');
  const selectEl = document.querySelector('.layout-1__select');
  if (!sidebar && !selectEl) return;

  const sidebarBtns = sidebar ? sidebar.querySelectorAll('.sidebar-nav__item') : [];
  const panels = document.querySelectorAll('.layout-1__panel');

  function showTopic(topicId) {
    // Hide all panels
    panels.forEach(panel => { panel.hidden = true; });
    // Deactivate all sidebar buttons
    sidebarBtns.forEach(btn => btn.setAttribute('aria-selected', 'false'));

    // Show selected panel
    const target = document.getElementById('layout1-' + topicId);
    if (target) target.hidden = false;

    // Activate sidebar button
    const activeBtn = sidebar ? sidebar.querySelector(`[data-controls="layout1-${topicId}"]`) : null;
    if (activeBtn) activeBtn.setAttribute('aria-selected', 'true');

    // Sync mobile select
    if (selectEl) selectEl.value = topicId;
  }

  // Sidebar button click handlers
  sidebarBtns.forEach(btn => {
    btn.addEventListener('click', () => {
      showTopic(btn.dataset.controls.replace('layout1-', ''));
    });
  });

  // Mobile select change handler
  if (selectEl) {
    selectEl.addEventListener('change', () => showTopic(selectEl.value));
  }
}
```

### JavaScript: initExpandableCards()

```javascript
// Source: W3C APG accordion pattern
function initExpandableCards() {
  const cards = document.querySelectorAll('.topic-card');
  if (!cards.length) return;

  cards.forEach(card => {
    const trigger = card.querySelector('.topic-card__trigger');
    const panel = card.querySelector('.topic-card__content');
    if (!trigger || !panel) return;

    trigger.addEventListener('click', () => {
      const isExpanded = trigger.getAttribute('aria-expanded') === 'true';
      const willExpand = !isExpanded;

      trigger.setAttribute('aria-expanded', String(willExpand));

      if (willExpand) {
        panel.hidden = false;
        // Force reflow before adding class (same pattern as existing fadeInPanel)
        void panel.offsetHeight;
        panel.classList.add('is-open');
      } else {
        panel.classList.remove('is-open');
        // Wait for animation before hiding
        if (!prefersReducedMotion()) {
          panel.addEventListener('transitionend', function handler() {
            panel.removeEventListener('transitionend', handler);
            panel.hidden = true;
          }, { once: true });
        } else {
          panel.hidden = true;
        }
      }
    });
  });
}
```

```css
  pointer-events: none; /* clicks handled on parent button */
}

/* aria-checked=false: first option active (Layout 1) */
.layout-toggle__btn[aria-checked="false"] .layout-toggle__option:first-child {
  background-color: var(--color-navy);
  color: var(--color-white);
}

/* aria-checked=true: second option active (Layout 2) */
.layout-toggle__btn[aria-checked="true"] .layout-toggle__option:last-child {
  background-color: var(--color-navy);
  color: var(--color-white);
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `role="tablist"` for sidebar nav | `<nav>` + `<button>` with standard Tab key | W3C APG 2023 clarification | Sidebar nav should NOT use the tab pattern; tabs require arrow-key roving focus which is unnatural vertically |
| JS `scrollHeight` measurement for accordion height | CSS `max-height` transition | CSS-Tricks documentation, widely adopted by 2020 | Removes 27+ lines of brittle JS |
| Custom checkbox/label for toggle | `<button role="switch" aria-checked>` | ARIA 1.1 specification | Correct semantics; Enter and Space work natively; no checkbox hacks |
| `position: fixed` + JS scroll listener for sidebar | `position: sticky` + CSS Grid `align-items: start` | CSS Grid widespread support, 2022+ | Zero JS needed for sidebar stickiness |
| `height: 0` to `height: auto` transition | `max-height: 0` to `max-height: 800px` | Long-established workaround | CSS fundamental: `auto` cannot be interpolated |

**Deprecated/outdated:**
- jQuery-based accordion plugins: Not needed; native CSS handles it
- `role="tablist"` on sidebar nav: Wrong ARIA semantics for sidebar navigation
- `overflow-y: scroll` on content area for independent scroll: Only needed if content area is height-constrained; this project uses normal page scroll which is preferable for accessibility

## Mobile Responsiveness

### Breakpoint Strategy (both layouts)

| Breakpoint | Layout 1 | Layout 2 |
|------------|----------|----------|
| < 640px (mobile) | Single column; sidebar hidden; `<select>` visible | Single column stack; cards full width |
| 640px-767px | Single column; still using `<select>` | Single column; cards full width |
| 768px+ (tablet/desktop) | Two-column grid activates; sidebar visible; `<select>` hidden | Single or 2-column grid (Claude's discretion) |
| 1024px+ (desktop) | Full sidebar + content layout | Cards at comfortable max-width |

### Mobile breakpoint recommendation (Claude's discretion item)

Use **768px** as the sidebar-to-select transition breakpoint (not 640px). Rationale: 768px is already the established nav breakpoint in this project, which means the 768px `@media` block already exists in `style.css` and adding Layout 1 responsive rules there keeps the breakpoints consistent. A 240px sidebar on a 640px tablet would be cramped (30% of width).

### Layout 2 desktop column recommendation (Claude's discretion item)

Use **single column** for Layout 2 on all sizes. Rationale: Topic cards contain substantial content (Get Started alone has 8+ items). A 2-column grid would create very tall and short cards side-by-side, harming scannability. Single column with generous padding matches how professional academic websites handle rich resource sections (confirmed by reviewing research.html and teaching.html patterns in this project).

## Layout Pattern Analysis

### Layout 1 — Sidebar Navigation: Strengths and Weaknesses

| Aspect | Assessment |
|--------|------------|
| Topic discoverability | HIGH on desktop (all 4 always visible). LOW on mobile (hidden in select). |
| Desktop space usage | 240px sidebar is 22% of 1100px container — acceptable, matches common documentation site convention |
| Implementation complexity | MEDIUM — requires adapting existing `switchTab()` plus mobile select sync |
| Accessibility | Well-supported by screen readers when using `<nav>` + `<button>` pattern; no exotic ARIA needed |
| Content density | Excellent for dense multi-section content; users can jump without scrolling |
| Comparison to Phase 4 tabs | Structurally similar but more scalable; tabs become crowded above 5 items, sidebar does not |

### Layout 2 — Expandable Cards: Strengths and Weaknesses

| Aspect | Assessment |
|--------|------------|
| Topic discoverability | HIGH on all sizes (all 4 card titles visible on load) |
| Mobile experience | Superior — natural single-column stack, consistent expand/collapse |
| Implementation complexity | LOW — simpler than Layout 1; no two-column grid, no mobile/desktop state sync |
| Accessibility | Well-supported; W3C APG accordion pattern is widely tested |
| Content density | Potentially tall page when multiple topics expanded; not a problem for single user comparing layouts |
| Comparison to Phase 4 tabs | More discoverable at a glance (titles always visible); tab panels completely hide other content |

### Which to default (Claude's discretion item)

**Show Layout 2 first by default** (HTML `data-layout="2"`). Rationale: Layout 2 has higher mobile discoverability and is simpler to evaluate immediately. The user wants to compare, so showing the more novel layout (cards) first draws attention to it, making the toggle to Layout 1 a deliberate action for comparison.

## Open Questions

1. **Sidebar active state on page load**
   - What we know: Layout 1 needs one topic panel visible by default; "Get Started" is first
   - What's unclear: Should "Get Started" be auto-expanded (visible) or require a click?
   - Recommendation: Set "Get Started" panel visible by default with `aria-selected="true"` on its sidebar button. Matches the tab interface precedent in Phase 4.

2. **Layout 2 — Should any card be pre-expanded on load?**
   - What we know: All 4 cards collapsed on load is the standard accordion default
   - What's unclear: Is it helpful to pre-expand "Get Started" so content is immediately visible?
   - Recommendation: Pre-expand "Get Started" only in Layout 2 (set `aria-expanded="true"` and remove `hidden`). This makes the page feel less empty on load and showcases the pattern's capability immediately.

3. **localStorage key conflict with potential future pages**
   - What we know: `localStorage.setItem('studentsLayout', '1')` is the planned key
   - What's unclear: No current conflict; this is the only page using layout toggles
   - Recommendation: Prefix the key: `'ibarron:studentsLayout'` — future-proof against collisions if other pages add similar toggles.
